    Displaying an object #

    The class Object #

    Java has a native class called Object with 11 instance methods. Every Java class implicitly extends Object, including user-defined classes. Therefore any Java class inherits these methods.

    The method toString #

    toString is one of the methods of the class Object. Here is (part of) its specification:

    /** Returns a string representation of the object.
 * In general, the toString method returns a string that "textually represents" this object.
 */
public String toString();

    This is the preferred way to display information about an object. For instance, the debugger of your IDE probably relies on this method to display a string that represents an object.

    This method is also called by many native methods and operators, such as System.out.println or concatenation with string (+ in Java).

    Example. These instructions

    Unicorn u = new Unicorn(3);
System.out.println(u);
String message = "Hello " + u;

    are equivalent to

    Unicorn u = new Unicorn(3);
System.out.println(u.toString());
String message = "Hello " + u.toString();

    Warning. In this example, System.out.println(u) does not display the reference held by the variable u (instead, it displays the output of u.toString()).

    More generally, the address of an object in memory cannot be displayed in Java (as opposed to the value of a pointer in C/C++).

    Overriding toString #

    The default implementation of toString (in the class Object) displays very little information about the object:

    public String toString() {
    return getClass().getName() + "@" +
      Integer.toHexString(hashCode());
}

    Warning. The output of hashCode() in this example is not the address of the object in memory.

    So a common practice consists in overriding toString.

    Example. Here is a possible implementation of the method toString within a class Unicorn:

    public class Unicorn {

    int xp;
    int moves;

    ...

    @Override
    public String toString() {
        return "Unicorn{" +
            "xp=" + xp + ", " +
            "moves=" + moves +
        "}";
    }
}

    Hint. Your IDE can generate such a method.

    Recursion #

    Warning. A naive implementation of toString (e.g. generated by an IDE) may have unexpected effects.

    Consider a version of our game where NPCs are located in rooms, some of which are adjacent to each other.

    The implementation may use the following class Room:

    public class Room {

    // characters present in the room
    NPC[] characters;
    // left adjacent room (null if there is none)
    Room leftAdjacentRoom;
    // similar to `leftAdjacentRoom`
    Room rightAdjacentRoom;

    // constructor
    public Room(NPC[] characters){
        this.characters = characters;
    }

    @Override
    public String toString() {
        return "Room{" +
            "characters=" + characters + ", " +
            "left=" + leftAdjacentRoom + ", " +
            "right=" + rightAdjacentRoom +
        "}";
    }
}

    Then a game layout with two adjacent rooms may be created as follows:

    NPC[] a1 = new NPC[]{ new Unicorn(3) };
Room r1 = new Room(a1);
NPC[] a2 = new NPC[]{ new Butterfly(7) };
Room r2 = new Room(a2);
r1.rightAdjacentRoom = r2;
r2.leftAdjacentRoom = r1;

System.out.println(r1);

    There is an issue with this program. Can you identify it?

    The program does not terminate, because toString is (implicitly) recursive, and each of the two rooms refers to the other.
