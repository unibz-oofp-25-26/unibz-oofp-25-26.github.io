    Encapsulation #

    Encapsulation is a (vague) principle in object-oriented programming that refers to “bundling” data with the code that operates on it, and restrict access to this code and data from other components of a system.

    From Wikipedia: “Essentially, encapsulation prevents external code from being concerned […]”

    Each component hides its internal logic by exposing only data and methods that other components may need.

    Good practice. A common approach consists in:

    • hiding all attributes and methods of a new class by default, and
    • relaxing this on a case-by-case basis, when needed.

    This is likely to be the default behavior of your IDE.

    Encapsulation also largely dictates how libraries are structured.

    Example. When we create a String in Java, we do not have access to the internal representation of the string object.

    in Java #

    Syntax. Each attribute or method of a class can have an access modifier, which specifies which other classes can access it.

    Example. The keywords private and protected below are access modifiers:

    private int myAttribute;

protected int myMethod(){
  return 1;
}

    Definition. Java supports four levels of access:

    • private restricts access to the current class,
    • “package-private” relaxes private by also allowing access from the folder of the current class (in Java, a folder for source code is called a package), excluding subfolders,
    • protected relaxes “package-private” by also allowing access from the subclasses of the current class,
    • public does not restrict access.

    Warning. There is no keyword for the “package-private” level.

    Instead, this is the default level for an attribute or method without access modifier.

    Example. The attribute myAttribute below is package-private:

    int myAttribute;

    Here is a recap table from the Oracle tutorials:

    keyword class package subclasses world
    private yes no no no
    none yes yes no no
    protected yes yes yes no
    public yes yes yes yes

    Warning. A method declared in an interface is (implicitly) public.

    Warning. If a method m1 overrides (or implements) a method m2, then m1 must be at least as accessible as m2 (otherwise the program does not compile).

    The following program does not compile. Can you see why, and how to fix this?

    ├── Run.java
└── npcs
    ├── NPC.java
    └── impl
        └── Unicorn.java

    public abstract class NPC {

    static String configFolder = "path/to/config";
}

    public class Unicorn extends NPC {

    String name;

    public Unicorn (String name){
        this.name = name;
    }

    public static String getConfigFilePath (){
        return configFolder + "/unicorn.properties";
    }
}

    public class Run {

    void testUnicorn(){
        Unicorn myUnicorn = new Unicorn("Storm");
        myUnicorn.name = "Tornado";
    }
}
    • Unicorn.getConfigFilePath tries to access the package-private attribute NPC.configFolder (it should be made protected of public),
    • Run.getConfigFilePath, tries to access the package-private attribute name of myUnicorn (it should be made public).

    Hint. Your IDE may suggest how to fix such compilation errors.

    To improve encapsulation, it is good practice to restrict access whenever possible (i.e. without compromising compilation).

    Good practice. In Java, we ca:

    • use private by default for all attributes and methods that we create, and
    • if the program does not compile, use an IDE to relax access to some of them.

    Encapsulation in this program can be improved. Can you see how?

    ├── Run.java
└── npcs
    ├── NPC.java
    └── impl
        └── Unicorn.java

    public abstract class NPC {

    public int xp;

    public NPC(int xp) {
        this.xp = xp;
    }

    public void meet(NPC other){
        xp += other.xp;
        other.xp = xp;
    }
}

    
public class Unicorn extends NPC {

    public Unicorn (){
        super(1);
    }

    @Override
    public void meet(NPC other){
        sleep();
        super.meet(other);
    }

    public void sleep(){
        xp += 1;
    }
}

    public class Run {

    void testUnicorn(){
        Unicorn u1 = new Unicorn();
        Unicorn u2 = new Unicorn();
        u1.meet(u2);
    }
}
    • NPC.xp can be made protected,
    • the constructor of NPC can be made protected,
    • NPC.meet can be made protected,
    • Unicorn.sleep can be made private.

    Note. The constructor of an abstract class can always be made protected (since it can only be called in the constructor of a subclass).

    Getters and setters #

    For attributes, the notion of “access” can be refined.

    An attribute may be:

    • neither visible nor modifiable, or
    • only visible, or
    • only modifiable, or
    • both visible and modifiable.

    This can be achieved with private attributes and so-called “getter” and “setter” methods.

    Example. In the following class, the attribute xp has public visibility but is not modifiable.

    public class Unicorn {

    private int xp;

    public int getXp(){
        return xp;
    }
}

    Example. In the following class, the attribute xp can be modified but is not visible.

    public class Butterfly {

    private int xp;

    public void setXp(int xp){
        this.xp = xp;
    }
}

    Hint. Getter and setter methods can be automatically generated by your IDE.

    Composition #

    Example (from Effective Java, Item 18).

    Consider a class MyHashSet that extends Java’s HashSet functionalities by keeping track of the number of times something has been added to the set (as opposed to the output of HashSet.size(), which returns the number of elements remaining in the set).

    This class myHashSet may have a private attribute int counter (initialized to 0) that keeps track of the number of elements added to the set so far. And the class may be implemented by overriding add and addAll in the expected way, i.e.:

    public class MyHashSet extends HashSet {

      private int counter;

      ...

      @Override
      public boolean add(E e){
          counter++;
          return super.add(e);
      }

      @Override
      public boolean addAll(Collection<? extends E> c){
          counter += c.size();
          return super.addAll(c);
      }
}

    However, this implementation of addAll would count every insertion twice, because the implementation of HashSet.addAll calls the method add (which is overridden in this case).

    A design pattern called composition can be used to avoid such unintended effects. Intuitively, instead of extending the original class, use an instance of the original class as a (private) attribute of the new class (e.g. an instance Hashset named set in this example).

    However, this requires re-implementing all methods of the original class (albeit in a straightforward way).

    Example.

    public class MyHashSet {

      public boolean isEmpty(){
            return set.isEmpty();
      }
}

    Prevent overriding or inheritance #

    As show by the example above, in order to improve encapsulation, one may want in some scenarios to forbid overriding a method or extending a class.

    Syntax. In Java, this can be enforced with the keyword final.

    Example.

    public final class NonExtensibleClass {
    ...
}

    public class MyClass{

  public final void nonOverridableMethod(){
        ...
  }
}

    Warning. In Java, a variable can also be declared final. But this has a different meaning.
