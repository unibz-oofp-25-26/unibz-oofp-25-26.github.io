    Pipeline #

    We will see in this section how functional languages provide a convenient syntax to compose functions that modify lists.

    This lets us apply a sequence of operations (filter, transform, aggregate, etc.) to a stream of incoming data, analogously to Unix-like pipes. This style of programming also pairs well with functors and monads, which we will see in later sections.

    Thanks to lazy evaluation, each operation in the pipeline can start feeding partial results to the next one, even if it has not finished processing its input list.

    Adoption. A variety of imperative languages (notably C++, C#, Java or Rust) have borrowed or adapted this pattern. For instance, pipelines have become an important part of Java programming, since the release of Java Stream API (2014).

    This pattern has also been adopted by some scripting languages for statistical analysis, such as R (natively) or Matlab (via libraries).

    In Haskell #

    Operations on a list #

    We have encountered several higher-order functions commonly used to specify operations on lists: map, filter, takeWhile, all, any, foldr, foldl, find, etc.

    Example. Recall the product type that we used to model a playing card:

    data Card = CardC { suit :: Suit, rank :: Int }

    Then the function

    filter (\c -> suit c == Spades)

    has type

    [Card] -> [Card]

    It retains all cards with suit Spades.

    Composition #

    Example (continued). The following function sums the ranks of all spades in a list of card.

    data Card = CardC { suit :: Suit, rank :: Int }

myPipeline :: [Card] -> Int
myPipeline = sum . map rank . filter (\c -> suit c == Spades)

    myPipeline is the composition of three functions:

    • the function filter (\c -> suit c == Spades) seen above,
    • the function map rank, which maps each retained card to its rank,
    • the function sum, which aggregates these ranks.

    Evaluation #

    Thanks to lazy evaluation, each function in a pipeline may process its input on the fly, without waiting for termination of the previous function in the pipeline.

    Example (continued). In the example above, let us use:

    • f for the first function in the pipeline (i.e filter (\c -> suit c == Spades)),
    • g for the second function in the pipeline (i.e map rank),
    • h for the third function in the pipeline (i.e sum).

    so that myPipeline can be defined as

    myPipeline = h . g . f

    For a list of cards cs, a naive evaluation strategy would:

    • first evaluate f cs,
    • then apply g to the resulting list,
    • the apply h to the resulting list.

    Instead, thanks to lazy evaluation, the function g can process cards on the fly, as soon as they are output by f (and similarly for h and g).

    Simplified syntax #

    Forward composition #

    Motivation. Consider a composition $c$ of functions.
    For instance

    $\qquad c = h \circ g \circ f$

    Computationally, this reads from right to left :

    $\qquad$ “apply $f$, then $g$, then $h$.”

    Instead, when we write code, it can be convenient to use an expression that mimics the order of execution.

    For instance, we could define a reverse composition symbol $\gtrdot$, and write:

    $\qquad c = f \gtrdot g \gtrdot h$

    Syntax. The Haskell module Control.Category provides a reverse composition function >>>.
    It can be defined as follows:

    (>>>) :: (a -> b) -> (b -> c) -> a -> c
f >>> g = g . f

    This lets us write

    $\qquad$ f >>> g >>> h

    instead of

    $\qquad$ h . g . f

    Example(continued). The pipeline above can alternatively be defined as

    import Control.Category

myPipeline :: [Card] -> Int
myPipeline =
  filter (\c -> suit c == Spades)
    >>> map rank
    >>> sum

    Let

    maybeHead :: [a] -> Maybe a

    map a nonempty list to its head (and the empty list to Nothing).

    Simplify the following function:

    myPipe :: [Int] -> Maybe Int
myPipe =
  filter even
    >>> map (+ 1)
    >>> filter even
    >>> maybeHead

    myPipe :: [Int] -> Maybe Int
myPipe _ = Nothing

    Forward application #

    A similar syntax can be used to chain function applications.

    Motivation.

    $\qquad h(g(f(x)))$

    means

    $\qquad$ “start from $x$, then apply $f$ to it, then apply $g$ to the result, and then apply $h$ to the result”.

    Instead, it can be more convenient to first write $x$, then $f$, then $g$, then $h$.

    Syntax. The module Data.Function provides a reverse application function &.
    It can be defined as:

    (&) :: a -> (a -> b) -> b
x & f = f x

    The function & associates to the left, so that x & f & g stands for (x & f) & g.

    This let us write

    $\qquad$ x & f & g & h

    instead of

    $\qquad$ h (g (f x))

    Example(continued). The pipeline above can alternatively be defined as

    import Data.Function

myPipeline :: [Card] -> Int
myPipeline cs =
  cs
    & filter (\c -> suit c == Spades)
    & map rank
    & sum

    Let

    maybeHead :: [a] -> Maybe a
maybeLast :: [a] -> Maybe a

    map a nonempty list to its head and last element respectively (and the empty list to Nothing).

    Evaluate the following expressions.

    1. [2,1,4,7] & filter even
          & map (+ 1)
          & maybeLast

    2.  [-5,4,2,-1] & filter (>= 0)
             & map (\x -> -x)
             & filter (> -2)
             & all (not . even)

    3.  [
   ("Bolzano", "BZ", 39100),
   ("Rovereto", "TN", 38068),
   ("Merano", "BZ", 39012),
   ("Bressanone", "BZ", 39042)
 ]
   & filter (\(_, prov, zip) -> prov == "BZ" && zip /= 39100)
   & map (\(name, _, _) -> name)
   & maybeHead
    1. Just 5
    2. True
    3. Just "Merano"

    in Java #

    Functional pipelines have been introduced in Java 8 (2014), via the interface Stream.

    Java Stream computations are specified as a pipeline that consists of:

    1. a source (e.g. a collection or an I/O channel),
    2. a sequence of zero or more intermediate operations (filter, map, etc.), each of which transforms a Stream into another,
    3. at most one terminal operation (e.g. collect the elements of the stream).

    Like in Haskell, a Java Stream pipeline may take advantage of lazy evaluation.

    Example. Consider the following classes NPC, Butterfly, Caterpillar and Unicorn,

    Here is a pipeline that transforms a list of NPCs into a set of integers:

    List<NPC> npcs = getNPCs();

Set<Integer> butterflyXPs = npcs.stream()                          // create a stream
                             .filter(c -> c instanceof Butterfly)  // retain butterflies
                             .map(c -> c.xp)                       // map each butterfly to its XP
                             .collect(Collectors.toSet());         // collect the values as a set

    The pipeline can be decomposed as follows:

    Source.

    npcs.stream()

    generates a stream out of the list npcs.
    The output has type Stream<NPC>.

    Intermediate operation.

    .filter(c -> c instanceof Butterfly)

    retains all instances of Butterfly in this stream.
    The output has type Stream<NPC>.

    Intermediate operation.

    .map(c -> c.xp)

    maps each remaining butterfly to its XP.
    The output has type Stream<Integer>.

    Terminal operation.

    .collect(Collectors.toSet());

    collects the XPs into a set.
    The output has type Set<Integer>.

    What do the following methods compute?

    Set<NPC> method1(Set<NPC> set1, Set<NPC> set2) {
    return set1.stream()
            .filter(c -> set2.contains(c))
            .collect(Collectors.toSet());
}

Set<NPC> method2(Set<NPC> set1, Set<NPC> set2) {
    return set1.stream()
            .filter(set2::contains)
            .collect(Collectors.toSet());
}

    Both methods compute the intersection of set1 and set2.

    What do the methods method1 to method4 below compute?

    Stream<NPC> method1(List<NPC> npcs) {
    return npcs.stream()
            .filter(c -> c.xp > 10);
}

List<NPC> method2(Stream<NPC> stream) {
    return stream.map(c -> transform(c))
            .toList();   // collects the stream as a final `List`
}

NPC transform(NPC c) {
    if (c instanceof Caterpillar) {
        return new Butterfly(c.moves);
    }
    return c;
}

List<NPC> method3(List<NPC> npcs) {
    return method2(method1(npcs));
}

List<NPC> method4(List<NPC> npcs) {
    return npcs.stream()
            .filter(c -> c.xp > 10)
            .map(c -> c instanceof Caterpillar ?
                    new Butterfly(c.moves) :
                    c
            )
            .toList();
}
    • method1:
      • Input: A list of npcs.
      • Output: a stream that contains the same npcs, but discarding npcs with less than 10 xp.
    • method2:
      • Input: A stream of npcs.
      • Output: A list that contains the same npcs, but where caterpillars become butterflies.
    • method3:
      • Input: A list of npcs.
      • Output: A list that contains the same npcs, but discarding npcs with less than 10 xp, and transforming caterpillars into butterflies.
    • method4 is equivalent to method3.

    In the rest of this section, we introduce a few common ways to specify the source, the intermediate operations and the terminal operation of a pipeline.

    Source #

    Collection #

    Syntax. The instance method

    Stream<E> stream()

    of the interface Collection<E> lets us create a stream out of a Collection.

    Hint. Recall that the interfaces List<E> and Set<E> both extend Collection<E>.

    Example.

    List<NPC> npcs = getNPCs();
Stream<NPC> myStream = npcs.stream();

    If the collection has a predictable order of iteration (e.g. if the collection is a List), then the elements will appear in this order in the stream.

    Hint. To create a stream out of the entries of a map, we can use the instance method entrySet of the interface Map, which we have seen earlier. It returns a Set (which is a collection).

    Example.

    Map<String, Integer> map = getMap();
Stream<Entry<String, Integer>> myStream = map.entrySet().stream();

    Array #

    Syntax. A stream can also be created out of an array, with the static method

    Stream<T> stream(T[] array)

    of the class Arrays.

    Example.

    NPC[] npcs = getNPCs();
Stream<NPC> myStream = Arrays.stream(npcs);

    Direct instantiation #

    Syntax. The static method

    Stream<T> of(T... values)

    of the interface Stream lets us instantiate a Stream directly with (a few) elements.

    Example.

    Stream<Integer> myStream = Stream.of(1,4,2,1);

    Intermediate operation #

    An intermediate operation transforms a stream into another.

    filter #

    Syntax. The instance method

    Stream<T> filter(Predicate<? super T> predicate)

    of the interface Stream<T> is analogous to Haskell’s filter, where ? super T stands for a supertype of T (possibly T itself).

    Reminder. Predicate<U> is a functional interface for Boolean methods (i.e. methods with type U $\to$ Boolean).

    Example.

    // contains [4,2,2]
Stream.of(1,4,2,1,2)
    .filter(x -> x % 2 == 0)

    map #

    Syntax. The instance method

    Stream<R> map(Function<? super T, ? extends R> function)

    of the interface Stream<T> is analogous to Haskell’s map, where:

    • ? super T stands for a supertype of T (possibly T itself), and
    • ? extends R stands for a subtype of R (possibly R itself).

    Reminder. Function<U,V> is a functional interface for methods with type U $\to$ V.

    Example.

    // contains [2,5,3,2,3]
Stream.of(1,4,2,1,2)
    .map(x -> x + 1)

    flatMap #

    Syntax. The instance method

    Stream<R> flatMap(Function<? super T, ? extends Stream<? extends R>> mapper)

    of the interface Stream<T> is analogous to Haskell’s concatMap.

    Example.

    // contains [1,2,4,5,2,3]
Stream.of(1,4,2)
    .flatMap(x -> Stream.of(x, x + 1))

    Example. Consider the following classes City and Region.

    We can transform a stream of regions into a “flat” stream that consists of all cities in these regions, as follows:

    Stream<Region> regions = getRegions();
Stream<City> cities = regions.flatMap(r -> r.cities.stream());

    In contrast,

    regions.map(r -> r.cities.stream())

    has type Stream<Stream<City>>.

    Example (continued). The following method maps a list of regions to the stream of zip codes of all cities in these regions, with the exceptions of the zip code(s) of Bologna:

    Stream<Integer> allZipCodesButBologna(List<Region> regions) {
    return regions.stream()
        .flatMap(r -> r.cities.stream())
        .filter(c -> !c.name.equals("Bologna"))
        .map(c -> c.zipCode);
}

    distinct #

    Syntax. The instance method

    Stream<T> distinct()

    of the interface Stream<T> produces an identical stream, but without duplicates.

    Example.

    // contains [2, 1, 3]
Stream.of(2, 1, 1, 3, 1, 2).distinct()

    For a stream of objects, duplicates are determined by the method equals (similarly to what we have seen already for a Set).

    Reminder. If a class overrides the method equals, then it should also override hashCode (and these two implementations should be consistent with each other).

    sorted #

    Syntax. The instance method

    Stream<T> sorted()

    of the interface Stream<T> produces an identical stream, but where elements are sorted. Its behavior is analogous to Arrays.sort and Collections.sort.

    To go further. An alternative version of sorted takes a Comparator as input.

    Example.

    // contains [1, 1, 2, 2, 3]
Stream.of(2, 1, 3, 1, 2).sorted()

    limit #

    Syntax. The instance method

    Stream<T> limit(long n)

    of the interface Stream<T> retains the n first elements of the stream (or the whole stream if its size if inferior to n).

    Example.

    // contains [2, 1, 3, 1]
Stream.of(2, 1, 3, 1, 2, 5).limit(4)

    skip #

    Syntax. The instance method

    Stream<T> skip(long n)

    of the interface Stream<T> discards the n first elements of the stream.

    Example.

    // contains [3, 1, 2]
Stream.of(2, 1, 3, 1, 2).skip(2)

    Stateless VS stateful intermediate operation #

    Terminology. A stateless operation processes each element of the input stream independently, without the need to memorize information about previous elements in the stream.

    An operation is stateful if it is not stateless.

    Examples.

    • filter, map and flatMap are stateless,
    • distinct, sorted, limit and skip are stateful.

    To go further. Stateless operations can be easily parallelized (as long as they have no side-effect), as opposed to stateful operations.

    Terminal operation #

    A stream pipeline has at most one terminal operation (and it must be the last one).

    Examples. A terminal operation may:

    • collect the elements of a stream (e.g. as a set), or
    • aggregate these elements (e.g. sum the values of a stream of integers), or
    • execute a method for each element (e.g. print its value).

    forEach #

    Syntax. The instance method

    void forEach(Consumer<? super T> action)

    of the interface Stream<T> applies the method action to each element of the stream.

    Reminder. Consumer<U> is a functional interface for methods with type U $\to$ void.

    forEach is similar to map, but its input method only operates via side-effects (since it has no return value).

    Example.

    // outputs `4, `2` and then `4` (each on a new line)
Stream.of(4, 1, 3, 2, 4)
      .filter(x -> x % 2 == 0)
      .forEach(System.out::println);

    allMatch and anyMatch #

    Syntax. The instance methods

    boolean allMatch(Predicate<? super T> predicate);

    and

    boolean anyMatch(Predicate<? super T> predicate);

    of the interface Stream<T> are analogous to the Haskell functions all and any

    Example.

    // has value `false`
boolean allEven = Stream.of(1, 3, 4, 2, 1)
          .allMatch(x -> x % 2 == 0);

    Example.

    // has value `true`
boolean anyEven = Stream.of(1, 3, 4, 2, 1)
          .anyMatch(x -> x % 2 == 0);

    Warning. anyMatch (resp. allMatch) is a short-circuiting operations, meaning that the stream is interrupted after the first match (resp. first non-match), if any.

    Example. The two following are equivalent:

    List<Integer> integers = getIntegers();
boolean anyEven = integers.stream()
          .anyMatch(x -> x % 2 == 0);

    $\qquad$

    List<Integer> integers = getIntegers();
boolean anyEven = false;
for(int x : integers){
    if(x % 2 == 0){
        anyEven = true;
        break;
    }
}

    Coding style. A method with multiple boolean flags can be complex to read and/or debug, in particular when combined with break and/or continue statements.

    Streams with short-circuiting operations (like anyMatch, allMatch and findFirst) can be a way to avoid them.

    Use anyMatch and/or allMatch to rewrite the following Java method:

    • without boolean flag, and
    • without break statement.
    /**
 * Returns true iff each of the input lists contains a number > 20
 */
boolean check(List<List<Integer>> lists) {

    boolean flag1 = true;
    boolean flag2;

    for (List<Integer> list : lists) {
        flag2 = false;
        for (int value : list) {
            if (value > 20) {
                flag2 = true;
                break;
            }
        }
        if (!flag2) {
            flag1 = false;
            break;
        }
    }
    return flag1;
}

    boolean check(List<List<Integer>> lists) {
    return lists.stream()
        .allMatch(l -> l.stream().anyMatch(v -> v > 20));
}

    Note. In this example, boolean flags and break statements can also be avoided with:

    • return statements, and
    • an auxiliary method.

    For instance:

    boolean check(List<List<Integer>> lists) {

    for (List<Integer> list : lists) {
        if(!isValid(list)) {
            return false;
        }
    }
    return true;
}

private boolean isValid(List<Integer> values) {
    for (int value : values) {
        if(value > 20) {
            return true;
        }
    }
    return false;
}

    findFirst #

    Syntax. The instance method

    Optional<T> findFirst()

    of the interface Stream<T> returns the first element of the stream, if any.

    It is analogous to the Haskell function maybeHead that we implement earlier (or equivalently, find (\x -> True)).

    Optional is the option type of Java (analogous to Haskell’s Maybe), which we will cover later on.

    Example.

    // has value `Optional[4]`
Optional<Integer> firstEven = Stream.of(1, 3, 4, 2, 1)
                    .filter(x -> x % 2 == 0)
                    .findFirst();

    Example.

    // has value `Optional.empty`
Optional<Integer> firstEven = Stream.of(1, 3, 1)
                    .filter(x -> x % 2 == 0)
                    .findFirst();

    Warning. findFirst is a short-circuiting operation (like allMatch and anyMatch).

    Example The following two programs are equivalent:

    List<Integer> integers = getIntegers();
Optional<Integer> firstEven = integers.stream()
                    .filter(x -> x % 2 == 0)
                    .findFirst();

    $\qquad$

    List<Integer> integers = getIntegers();
Optional<Integer> firstEven = Optional.empty();
for(int x : integers){
    if(x % 2 == 0){
        firstEven = Optional.of(x);
        break;
    }
}

    count #

    Syntax. The instance method

    int count()

    of the interface Stream<T> returns the size of the stream.

    Example.

    // has value 2
int size = Stream.of(2, 1, 3, 1, 2)
            .filter(x -> x % 2 == 0)
            .count();

    max #

    Syntax. The instance method

    Optional<T> max(Comparator<? super T> comparator);

    of the interface Stream<T> returns the (first) maximal element in the stream (if any) according to comparator.

    In this declaration:

    • Optional is the option type of Java (analogous to Haskell’s Maybe), which we will cover later on.

    • Comparator<U> is a functional interface that stands for the type U $\times$ U $\to$ Integer.

      A method that implements Comparator<U> specifies a sorting criterion (aka total preorder) overT, similarly to the method compareTo.

    Example.

    // has value `Optional[4]`
Optional<Integer> max = Stream.of(2, 1, 3, 4, 5)
          .filter(x -> x % 2 == 0)
          .max(Integer::compare);

    Example.

    // has value `Optional.empty`
Optional<Integer> max = Stream.of(5, 1)
          .filter(x -> x % 2 == 0)
          .max(Integer::compare);

    Example.

    City ancona = new City("Ancona", 60100);,
City bologna = new City("Bologna", 40100);,
City matera = new City("Matera", 75100);,

// has value `Optional[matera]`
Optional<City> selectedCity = Stream.of(ancona, bologna, matera)
                .max((c1, c2) -> c1.zipCode - c2.zipCode);

    reduce #

    Syntax. The instance method

    T reduce(T defaultValue, BinaryOperator<T> function);

    of the interface Stream<T> is a simplified left fold, where:

    • defaultValue is the default value (for the empty Stream), and
    • function is the aggregation function.

    Reminder. BinaryOperator<T> is a functional interface for methods with type T $\times$ T $\to$ T).

    Example.

    // has value 24
int product = Stream.of(1, 2, 3, 4)
                .reduce(1, (x, y) -> x * y);

    To go further. For more complex folds, the interface Stream provides alternative versions of reduce.

    toList #

    Syntax. The instance method

    List<T> toList()

    of the interface Stream<T> produces a final List with the elements of the stream.

    Warning. This method was introduced in Java 16. You cannot use it in your assignments, because Codeboard uses Java 8.

    collect #

    Syntax. The instance method collect of the interface Stream<T> takes as argument a so-called Collector, which is in charge of collecting the elements of the stream into:

    • a Collection (e.g. List or Set), or
    • a Map, or
    • a String,
    • etc.

    The utility class Collectors provides a variety of static methods that return a Collector. We highlight below a few of them

    Collectors.toSet #

    This collector produces a Java Set out of the elements of current stream.

    Example.

    // contains {1, 2, 3}
Set<Integer> mySet = Stream.of(2, 1, 3, 1, 2)
              .collect(Collectors.toSet());

    Note. The Java specification does not require a specific implementation. So this set could for instance be a HashSet or a TreeSet, depending on compiler and/or execution environment.

    Collectors.toList #

    This collector is analogous to Collectors.toSet, but produces a Java List.

    Example.

    // contains [2, 1, 3, 1, 2]
List<Integer> myList = Stream.of(2, 1, 3, 1, 2)
              .collect(Collectors.toList());

    Note. As for Collectors.toSet, there is no requirement on the specific implementation of this List.

    Collectors.joining #

    This collector concatenates a stream of strings.

    Syntax. Collectors.joining (optionally) take a String as input, which becomes the separator in the output string.

    Example

    City ancona = new City("Ancona", 60100);
City bologna = new City("Bologna", 40100);
City matera = new City("Matera", 75100);

// contains the string `Ancona:Bologna:Matera`
String concat = Stream.of(ancona, bologna, matera)
        .map(c -> c.name)
        .collect(Collectors.joining(":"));

    To go further. Other useful collectors include:

    • Colllectors.toMap
    • Colllectors.groupingBy
    • Colllectors.toCollection (for a specific implementation of a Collection)
