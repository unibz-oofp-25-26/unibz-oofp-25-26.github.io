    Monad #

    Like functors, monads come from category theory.

    For programmers, a monad can be viewed as a functor that implements additional functions, while respecting certain constraints.

    In Haskell #

    We have seen in the previous section that a functor is a parameterized type that implements the function fmap.

    A monad is a functor that implements two additional functions, called return and join.

    Example. We have seen how the parameterized type “List” (written [ ] in Haskell) can be made a functor by implementing fmap.

    We will see that it can also be made a monad, by implementing return and join.

    Notation. We will use m in this section to refer to a Haskell monad.

    Core functions #

    return #

    The function

    return :: a -> m a

    specifies how to produce an element of type m a out of a single element of type a.

    Example (continued). For the monad “List”, return wraps an element into a singleton list.

    It can be defined as

    return :: a -> [a]
return x = [x]

    join #

    The function

    join :: m (m a) -> m a

    “flattens” an m (m a) into an m a.

    Example (continued). For the monad “List”, join concatenates the inner lists of a list of lists.

    It can be defined as

    join :: [[a]] -> [a]
join xss = [ x | xs <- xss, x <- xs ]

    Concrete illustration.

    data City = Bolzano | Merano | Trento | Rovereto
  deriving (Show) -- only needed for printing

main :: IO ()
main = print $
    join [[Bolzano, Merano], [Trento, Rovereto]]

    In this program, the expression

        join [[Bolzano, Merano], [Trento, Rovereto]]

    flattens its input list, by concatenating the two inner lists.

    So the program prints

    [Bolzano, Merano, Trento, Rovereto]

    Derived functions #

    Two useful higher-order functions can be derived from join and fmap (recall that a monad is a functor, so it implements fmap). Both are infix functions, noted >>= and >=> respectivbely.

    >>= (pronounced “bind” or “flatmap”) #

    The function >>= has type

    (>>=) ::  m a -> (a -> m b) -> m b

    Let x :: m a and f :: (a -> m b).
    Then the evaluation of x >>= f:

    • applies f to each a “wrapped” inside x, and
    • “flattens” (with join) the resulting m (m b) into an m b.

    Example (continued). For the monad “List”, the function >>= can be defined as follows:

    (>>=) :: [a] -> (a -> [b]) -> [b]
xs >>= f = [ y | x <- xs, y <- f x ]

    Concrete illustration (continued). Let us extend the program above with a type Province and a function cities, as follows:

    data Province = AltoAdige | Trentino
data City = Bolzano | Merano | Trento | Rovereto
  deriving (Show) -- only needed for printing

cities :: Province -> [City]
cities AltoAdige = [Bolzano, Merano]
cities Trentino = [Trento, Rovereto]

    And modify the function main, as follows:

    main :: IO ()
main = print $
      [AltoAdige, Trentino] >>= cities

    The expression

          [AltoAdige, Trentino] >>= cities

    applies the function cities to each of the two provinces, and flattens the resulting list.

    So the program prints

    [Bolzano, Merano, Trento, Rovereto]

    Observe that this is a flat list (rather than a list of lists).

    Derivation. The function >>= need not be implemented if join and fmap already are. It can be derived as

    (>>=) :: m a -> (a -> m b) -> m b
x >>= f = join (fmap f x)

    Alternative definition of a monad. We can also implement >>= and derive join from it, as follows:

    join :: m (m a) -> m a
join x = x >>= id

    So there is an alternative (equivalent) way to define a monad, where >>= is primitive and join is derived from it.

    You will find this alternative definition in some textbooks or websites.

    Usage of >>= VS fmap VS function application #

    Recap. The following table recaps when >>=, fmap and plain function application can be used, for some function f and expression x (depending on their types).

    For instance, the first line reads as follows:

    $\qquad$ if x has type a and f has type a -> b, then f x has type b.

    type of $x$ type of $f$ expression $e$ type of $e$
    a a -> b f x b
    m a a -> b fmap f x m b
    m a a -> m b x >>= f m b

    Consider the following function:

    preds :: Int -> [Int]
preds x = [ y | y <- [0, 1 ..], y < x ]

    Complete the table below

    expression is valid value (if valid)
    preds [3] no
    preds 3 yes [0,1,2]
    preds 0
    3 >>= preds
    [3] >>= preds
    return 3 >>= preds
    [[3]] >>= preds
    [2,3] >>= preds

    fmap preds [2, 3]
    expression is valid value (if valid)
    preds [3] no
    preds 3 yes [0,1,2]

    return 3 >>= preds yes [0, 1, 2]
    [[3]] >>= preds no

    fmap preds 3 no
    fmap preds [2, 3] yes [[0, 1], [0, 1, 2]]

    >=> (pronounced “fish”) #

    Derivation. The function >=> is best explained by its derivation from >>=:

    (>=>) :: (a -> m b) -> (b -> m c) -> (a -> m c)
(f >=> g) x = f x >>= g

    Syntax. the parentheses in (f >>= g) x are unnecessary. We only added them for clarity.

    Intuition. The function >=> “chains” two functions

    f :: a -> m b
g :: b -> m c

    into a function

    f >=> g :: a -> m c

    This function f >=> g applies (>>= g) to the output of f.

    Example (continued). For the monad “List”, the function >=> can equivalently be defined as:

    (>=>) :: (a -> [b]) -> (b -> [c]) -> a -> [c]
(f >=> g) x = [ y | xs <- f x, y <- g xs ]

    Concrete illustration (continued). Let us extend the program above with a type Region and a function provinces, as follows:

    data Region = TrentinoAltoAdige
data Province = AltoAdige | Trentino
data City = Bolzano | Merano | Trento | Rovereto
  deriving (Show) -- only needed for printing

provinces :: Region -> [Province]
provinces TrentinoAltoAdige = [AltoAdige, Trentino]

cities :: Province -> [City]
cities AltoAdige = [Bolzano, Merano]
cities Trentino = [Trento, Rovereto]

    And modify the function main, as follows:

    main :: IO ()
main = print $
    (provinces >=> cities) TrentinoAltoAdige

    The expression

        (provinces >=> cities) TrentinoAltoAdige

    is equivalent to

        provinces TrentinoAltoAdige >>= cities

    So the program still prints

    [Bolzano, Merano, Trento, Rovereto]

    Note. The relation between >>= and >=> is analogous to the relation between function application and function composition (or between their reversed versions & and >>>).

    Usage. In some places, >=> can be more convenient than >>=, in the same way that function composition (e.g. g . f) can be more convenient than function application (e.g. g (f x)).

    Concrete illustration (continued). The function definition

    extractCities :: Region -> [Cities]
extractCities r = provinces r >>= cities

    can be simplified as

    extractCities :: Region -> [Cities]
extractCities = provinces >=> cities

    Constraints #

    We have seen previously that the function fmap must satisfy additional constraints for a parameterized type to be a valid functor.

    Similarly, the functions implemented by a monad must satisfy certain constraints.

    One can formulate these constraints by focusing on the >=> and return functions, as follows:

    Constraints. A monad must satisfy two constraints:

    • >=> is associative, i.e.

    $\qquad$ (f >=> g) >=> h = f >=> (g >=> h)

    • return is a neutral element for >=>, i.e.

    $\qquad$ f >=> return = return >=> f = f

    Alternative formulations. There are several common (equivalent) ways to formulate the constraints that a monad must satisfy. In particular, they can be expressed as constraints over join or >>=, rather than >=> (as we did here).

    You may find these alternative definition in some textbooks or websites.

    Maybe as a monad #

    We have seen above how the functor “List” (noted [ ] in Haskell) can be made a monad, by implementing the functions return an join.

    We will now do this with the functor Maybe.

    As a reminder, Maybe is declared in the Haskell Prelude as:

    data Maybe a = Nothing | Just a

    Core functions #

    return :: a -> m a #

    The function return for the type Maybe simply wraps a value (with the constructor Just):

    return :: a -> Maybe a
return x = Just x
    join :: m (m a) -> m a #

    The function join for the type Maybe transforms a Maybe (Maybe a) into a Maybe a in the expected way:

    join :: Maybe (Maybe a) -> Maybe a
join Just (Just x) = Just x
-- covers the two remaining cases: `Nothing` and `Just Nothing`
join _ = Nothing

    Derived functions #

    Bind, a.k.a. >>= :: m a -> (a -> m b) -> m b #

    The function >>= for the type Maybe can be defined as

    (>>=) :: (Maybe a) -> (a -> Maybe b) -> Maybe b
-- propagate `Nothing`
Nothing >>= _ = Nothing
-- otherwise apply the function
Just x >>= f = f x

    Equivalently, it can be derived from join and fmap, as explained above.

    The function >>= is convenient to chain computations that may “fail” (i.e. return Nothing): it lets us write code that automatically takes care of propagating failures (a.k.a. “null values”).

    Example. Consider the two following functions

    safeDiv :: Int -> Int -> Maybe Int
safeDiv _ 0 = Nothing
safeDiv x y = Just (x / y)

maybeLog :: Int -> Maybe Float
maybeLog x | x > 0 = Just (logBase 2 x)
           | otherwise = Nothing

    Intuitively,

    • safeDiv only “succeeds” for a divisor $\neq 0$, and
    • maybeLog only “succeeds” for an input $> 0$.

    The function >>= lets us feed the output of safeDiv to maybeLog, without the need to check whether safeDiv succeeded.

    For instance:

    • safeDiv 24 3 >>= maybeLog evaluates to 3 (because safeDiv and maybeLog both “succeed”)
    • safeDiv (-24) 3 >>= maybeLog evaluates to Nothing (because maybeLog “fails”)
    • safeDiv 4 0 >>= maybeLog also evaluates to Nothing (because safeDiv “fails”, and >>= propagates the failure).

    Recall the following functions:

    • even :: Int -> Bool, which maps an integer to True iff it is even,
    • filter :: (a -> Bool) -> [a] -> [a], where filter f xs retains the elements of xs for which f evaluates to True,
    • maybeLast :: [a] -> Maybe a, which maps a nonempty list its last element (and the empty list to Nothing),
    • maybeLog :: Int -> Maybe Float from our the example immediately above.

    Now consider the following pipeline function:

    myPipe :: [Int] -> Maybe Float
myPipe xs = xs & filter even
               & maybeLast
               >>= maybeLog

    Evaluate the following expressions

    • myPipe []
    • myPipe [8]
    • myPipe [3,5]
    • myPipe [4,3,-2,5]
    myPipe :: [Int] -> Maybe Float
myPipe xs = xs & filter even
               & maybeLast
               >>= maybeLog
    • myPipe [] evaluates to Nothing (maybeLast “fails” and >>= propagates Nothing)
    • myPipe [8] evaluates to Just 3 (maybeLast and maybeLog both “succeed”)
    • myPipe [3,5] evaluates to Nothing (maybeLast “fails” and >>= propagates Nothing)
    • myPipe [4,3,-2,5] evaluates to Nothing (maybeLast “succeeds” and maybeLog “fails”)
    Fish, a.k.a >=> :: (a -> m b) -> (b -> m c) -> a -> m c #

    Example. Using >=> and forward function composition (>>>), we can rewrite the pipeline of the exercise immediately above in a more concise fashion:

    myPipe :: [Int] -> Maybe Float
myPipe = filter even
          >>> maybeLast
          >=> maybeLog
