    Option type #

    We introduce an important generic and composite type, available in many languages, under a variety of names (Maybe, Option, optional, Optional, Nullable, etc.)

    We will see in the dedicated sections that such a type can also be made a functor, and (perhaps most importantly) a monad. For now, we will use it without these extra features.

    Motivation #

    Problem. A programming language may not have a type that exactly fits the domain of a function that we want to implement.

    Examples. A programming language may not have a type that exactly fits:

    • positive integers,
    • integers $\neq 0$,
    • ASCII characters,
    • nonempty lists (or nonempty arrays, nonempty sets, etc.),
    • pairs of distinct elements,
    • phone numbers,
    • invertible matrices,
    • etc.

    Consequence. When we implement a function, we may need to handle inputs that should not be in its domain.

    Examples.

    • A function that returns the head of a list may receive the empty list as input,
    • a function that divides two numbers may receive 0 as input divisor,
    • a function that maps a date to the average temperature in Bolzano may receive a future date as input,
    • etc.

    null value and propagation #

    A common way for a function/method $f$ to handle non-supported inputs (without throwing an exception) consists in returning a special value. Depending on the language, this value may be null, nil, undefined, etc.

    Consequence. A program that calls $f$ must take this scenario into consideration:

    • if $f$ returns null, take a special action (e.g. return null as well),
    • otherwise proceed normally.

    Drawback. Programmers often forget to check whether a value is null.

    Indeed, in most programming languages, the return type of $f$ does not indicate that the value may be null (for instance, in Java, a method with return type Integer may return null).

    Non-handled null values are a notorious source of bugs (“null pointer exception”, “segmentation fault”, etc.).

    Second drawback. Handling null values explicitly creates boilerplate code (made worse by the fact that handling a null value often consists in returning one).

    Option type #

    Functional languages typically do not have null values. Instead, they rely on a special type (or sometimes several) for values that may be null.

    These are often called option type or maybe type.

    Benefits.

    1. An option return type for a function $f$ signals that it may not return a value. So this case must be handled by the code that is calling $f$.

    2. Option types can be paired with higher-order functions that propagate “missing values” automatically, thus reducing boilerplate code. We will introduce such functions in the sections on functors and monads

    in Haskell #

    Haskell uses a parameterized option type called Maybe.

    Declaration. An expression with type Maybe a may evaluate to either nothing, or a value of type a (where a is a type variable).

    Accordingly, the type Maybe is declared in the Haskell Prelude as

    data Maybe a = Nothing | Just a

    Example. The following function maps a nonempty list to its head, and the empty list to nothing.

    maybeHead :: [a] -> Maybe a
maybeHead [] = Nothing
maybeHead (x : _) = Just x

    Implement the function maybeLast :: [a] -> Maybe a that maps a nonempty list to its last element, and the empty list to Nothing.

    maybeLast :: [a] -> Maybe a
maybeLast [] = Nothing
maybeLast [x] = Just x
maybeLast (_ : xs) = maybeLast xs

    Implement the native function find :: (a -> Bool) -> [a] -> Maybe a so that

    find f xs evaluates to:

    • the first element of xs for which f evaluates to True, and
    • Nothing if there is no such element.
    find :: (a -> Bool) -> [a] -> Maybe a
find _ [] = Nothing
find f (x : xs) | f x = Just x
                | otherwise = find f xs
