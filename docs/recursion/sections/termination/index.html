    Recursion and termination #

    Tail recursive function #

    In Haskell #

    Exercise.

    The function failedReverse below seems similar to the function reverse seen in the previous section.

    However, there is something wrong with it. Can you identify what it is?

    failedReverse :: [Int] -> [Int]
failedReverse [] = []
failedReverse (x : xs) = failedReverse (xs ++ [x])

main :: IO ()
main = print $ failedReverse [1,2,3]

    Solution.

    The recursive call (Line XXX) is performed on a list that is not smaller than the input list.

    As a consequence, if ys is a nonempty list, then the evaluation of failedReverse ys does not terminate.

    Execution. The function failedReverse is tail-recursive. The compiler is able to transform it into a loop, which in this case does not terminate. So if we execute the program, it simply loops, indefinitely.

    Stack overflow #

    In Haskell #

    Example. Here is another non-terminating program

    failedId :: Int -> Int
failedId 0 = 0
failedId x = failedId (x + 1) - 1

main :: IO ()
main = print $ failedId 3

    Note. As opposed to the previous example, the function failedId here is not tail-recursive. The compiler is unable to transform it into a loop.

    As a consequence, the call stack grows as follows

    ... etc

failedId 8
failedId 7
failedId 6
failedId 5
failedId 4
failedId 3
main

    until its capacity is exhausted. When this happens, the program is interrupted with a stack overflow exception.

    Definition. A stack overflow exception is thrown during the execution of a program if the capacity of the call stack is exhausted.

    Warning. A stack overflow is usually due to a non-terminating recursive call. However, a non-terminating recursive call may not result in a stack overflow, as illustrated with our first example.

    Lazy evaluation #

    In Haskell #

    Example. Recall the function take :: Int -> [a] -> [a] seen the previous section, where take k xs consists of the first k elements of the list xs (or the whole xs if it has less than k elements).

    Now consider the following program

    succs :: Integer -> [Integer]
succs x = x : succs (x + 1)

main :: IO ()
main = print $ take 5 (succs 3)

    Execution.

    Intuitively, succs 3 is the (infinite) list of integers $\ge 3$, i.e. the list $3, 4, 5, 6, \ldots $

    A naive evaluation strategy for the expression take 5 (lazy 3) would proceed in two steps:

    1. first evaluate succs 3, and
    2. then retain the first 5 elements of the resulting list.

    Since the evaluation of succs 3 (a.k.a. Step 1) does not terminate, the whole program would not terminate either.

    Instead, Haskell’s lazy evaluation allows a different strategy, where only the first 5 elements of lazy 3 are computed.

    As a consequence, the program terminates and prints [3,4,5,6,7].

    Example (continued).

    Let us simplify the function main as follows:

    main :: IO ()
main = print $ succs 3

    Execution.

    As expected, this new program does not terminate. However, thanks to lazy evaluation, the program can still print the elements succs 3 on the fly, as soon as thy are computed. This is analogous to processing a continuous stream of incoming data.

    Infinite lists #

    Several built-in Haskell functions produce infinite lists (like the function succs above). They can be convenient to define functions that act on the $n$ first elements of a list, when $n$ is not known in advance.

    Example. The Haslell Prelude provide the function iterate :: (a -> a) -> a -> [a], where iterate f x is the infintie list of iterative applications of f to x

    For instance iterate (+1) x is equivalent to the function succs above.

    Exercise.

    Define the function repeat :: a -> [a] that repeats an elemtn infinitely many times.

    Possible solution.

    repeat :: a -> [a]
repeat = iterate id

    Alternative solution.

    repeat :: a -> [a]
repeat x = x : repeat x
