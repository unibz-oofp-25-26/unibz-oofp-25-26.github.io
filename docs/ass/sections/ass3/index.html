    Exercise 1: colored lists #

    We consider lists that:

    • consist of green and red nodes, where
    • each node is labeled with an integer, and
    • the head node is green.

    We model such lists with the following classes and interfaces:

    Restriction. You can assume that lists are acyclic, i.e. no node points to one of its ancestors.

    Complete or modify the abstract class AbstractNode and the classes RedNode, GreenNode and HeadNodeImpl in the folder ex1, so that:

    • RedNode and GreenNode correctly implement the interface Node, and
    • HeadNodeImpl correctly implements the interface HeadNode.

    The expected behavior of each method is specified in the corresponding interface.

    Hint. Feel free to:

    • create auxiliary methods to simplify your code, and/or
    • use the abstract class (called AbstractNode) to avoid duplicating your own code, and/or
    • extend the interfaces with additional methods.

    Hint. Some methods may be easier to implement recursively.

    Hint. You can debug your code by running the tests in the folder test/java/ex1.

    Convention. The log for a failing test may mention the position of a node in a list. The head of a list has position 1.

    Exercise 2: card bots #

    We consider a game where each player holds a hand of cards.

    The table carries a pile of cards.

    Terminology. The card on top of the pile it called the visible card.

    On their turn, players can either:

    1. Swap the visible card with a card in their hand, and end their turn, or
    2. play cards from their hand by putting them on top of the pile, one by one (so that each card put on the pile becomes the visible card).

    A card can be played if it has either:

    • the same suit as the visible card and a superior rank (where “ace” is the lowest rank and “king” is the highest), or
    • the same rank as the visible card and a superior suit (in alphabetical order, i.e. Clubs -> Diamonds -> Hearts -> Spades).

    The goal for a player is to discard as many cards as possible.

    Restriction. You can assume that the hand of a player before a turn is nonempty (otherwise the game would be over).

    Complete or modify the abstract class Bot and the classes SuitFirstBot and RankFirstBot in the folder ex2, so that the two latter correctly implement the interface Player, with the behaviors described below.

    Hint. Feel free to:

    • create auxiliary methods to simplify your code, and/or
    • use the abstract class Bot to avoid duplicate code.

    Hint. You can debug your code by running the tests in the folder test/java/ex2.

    Playing a card #

    SuitFirstBot #

    This bot tries to play a card with lowest possible suit, and then lowest possible rank among those.

    RankFirstBot #

    This bot tries to play a card with lowest possible rank, and then lowest possible suit among those.

    Swapping a card #

    If a bot cannot play any card when its turn starts, then it swaps the visible card with the first (a.k.a. leftmost) card in its hand.

    Order of cards in hand #

    Within a hand, cards are sorted:

    • by suit (in alphabetical order), and
    • by rank within the same suit.

    Restriction. You can assume that the hand of a bot is already sorted when a method of the interface Player is called.

    Constraint. After executing one of these methods, the hand of a bot must still be sorted. In particular, when a swap is performed, the visible card must be inserted into the bot’s hand at the correct position.

    Syntax. In Java, two values v1 and v2 for an enumerated type can be compared with the native instance method compareTo. This method returns:

    • a negative int if v1 is declared before v2 (in the definition of the enumerated type),
    • 0 if v1 and v2 are the same value,
    • a positive int if v1 is declared after v2 (in the definition of the enumerated type).

    Example. Consider the following enumerated type:

    enum MyEnum {VALUE1, VALUE2}

    Then

    VALUE1.compareTo(VALUE2) < 0

    evaluates to true.
