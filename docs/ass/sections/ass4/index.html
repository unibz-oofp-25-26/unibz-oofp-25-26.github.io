    Assignment 4: abstract data types #

    Exercise 1: board game #

    We model a simple board game with rooms and players.

    Each room has:

    • a set of adjacent rooms, and
    • a set of players present in the room.

    Each player has

    • a type (paper, rock or scissor), and
    • a number of health points.

    An action consists in moving one player from a room to another (adjacent) room:

    • When the player enters the room, it may deal one point of damage to each other player that is already in the room (based on their rock/paper/scissor types), and receive damage in the same way.
    • After all these operations have been performed, if a player has no more health points, then it should be removed from the board.

    Complete the class BoardImpl so that it correctly implements the interface Board. The expected behavior of each method is described in the interface.

    Hint. Feel free to create auxiliary (private) methods to simplify your code.

    Hint. You can debug your code by running the tests in the folder test/java/ex1.

    Exercise 2: restaurant #

    We model arrivals, departures and waiting line in a restaurant.

    A restaurant has a fixed number of tables, each with a certain capacity (i.e. the number of people who can sit at this table).

    When a party arrives:

    • if there is an empty table, then the party gets immediately seated at (one of) the smallest table(s) that can host it,
    • otherwise the party is put on the waiting list.

    When a party finishes eating, it pays and leaves the restaurant, thus freeing up a table for another party to sit. When this happens, the next party to be seated should be the first on the waiting line that fits at this table.

    A party on the waiting list may also decide to leave before being seated.

    Complete the class RestaurantImpl so that it correctly implements the interface Restaurant. The expected behavior of each method is specified in the interface.

    Hint. Make use of the abstract data types seen during the lectures: List, Set, Map, etc.

    For instance, you may maintain (among others) a Java Map that maps each table capacity to the number of free tables that have this capacity.

    Hint. Aside from the methods seen in the lectures, the following may simplify the exercise:

    • List.remove(Object o)
    • TreeMap.ceilingKey(k key) or TreeMap.ceilingEntry(k key)

    Hint. Feel free to create auxiliary methods to make your code more readable and/or easier to maintain.

    Hint. You can debug your code by running the tests in the folder test/java/ex2.
