    Programming paradigms #

    A programming paradigm is a way to write programs. Programming languages can be categorized based on the paradigms (and sub-paradigms) that they support. For instance:

    • imperative programming,
    • object-oriented programming,
    • functional programming,
    • logical programming,
    • constraint programming,
    • etc.

    Warning. The boundaries of a paradigm are not always clear, and can be subjective.

    For this reason, we will (cautiously) borrow descriptions from Wikipedia.

    Imperative programming #

    The most popular programming languages traditionally support the imperative (sometimes called procedural) paradigm.

    Examples. Imperative languages include (in alphabetical order):

    • C/C++,
    • C#,
    • Go,
    • Java,
    • Javascript,
    • Kotlin,
    • Lua,
    • Perl,
    • PHP,
    • Python,
    • Rust,
    • Ruby,
    • Scala,
    • Typescript,
    • Visual Basic,
    • …

    From Wikipedia:

    “Imperative programming focuses on describing how a program operates step by step […]. The term is often used in contrast to declarative programming, which focuses on what the program should accomplish without specifying all the details of how the program should achieve the result.”

    Warning. The imperative vs declarative (a.k.a. “how” vs “what”) distinction is not binary, but lies on a spectrum: some languages or programs are more imperative/declarative than others.

    Features. Imperative languages tend to share similar:

    • elementary operations (variable assignment, etc.)
    • control structures (if/then/else statements, loops, method calls, etc.),
    • data structures (arrays, linked lists, associative arrays, etc.)

    Pseudocode is often used to describe imperative algorithms, in a language independent fashion.

    Object-oriented programming (OOP) #

    Object-oriented languages are (usually) also imperative.

    Examples. Object oriented languages include:

    • C++,
    • C#,
    • Java/Kotlin/Scala,
    • Ruby,
    • Visual Basic,
    • …

    Besides, many imperative languages provide some object-oriented features: Rust, Javascript/Typescript, Python, etc.

    From Wikipedia:

    “An OOP computer program consists of objects that interact with one another.”

    OOP also lets programmers declare classes, which are blueprints for sets of objects. Besides, many languages allow a class to extend another, this inheriting some of its features.

    This allows structuring code in a modular and readable way, while avoiding code duplication.

    OOP is widely used to structure moderate to large sized projects.

    Java #

    Java is one of the most popular object-oriented languages.

    Java’s syntax is influenced by C++ (albeit without pointers and explicit memory allocation), and C# was in turn heavily influenced by Java.

    Java (like C++ and C#) has been extended with functional features over the years, notably with the release of Java 8 (2014).

    In this course, we will use Java 1.21 (more commonly called Java 21).

    Functional programming #

    Functional languages are less common than imperative ones. They include:

    • Elm,
    • Erlang,
    • Haskell,
    • Lisp dialects (Common Lisp, F#, OCaml, etc.)
    • ML dialects (Clojure, Scheme, Standard ML, etc.)
    • PureScript
    • Scala,
    • etc.

    However, some functional features are key to popular imperative languages:

    • callback methods in Javascript,
    • list comprehensions in Python,
    • stream-like
    • predefined higher-order functions (map, filter, reduce, etc.) for data processing in Rust and modern C++, C# and Java,
    • immutability,
    • etc.

    From Wikipedia:

    “Functional programming has its roots in academia, evolving from the lambda calculus, a formal system of computation based only on functions.”

    The functional paradigm, “declares the desired result as the value of a series of function evaluations”, and “avoids state and mutable data.”

    “Proponents of purely functional programming claim that by restricting side effects, programs can have fewer bugs, be easier to debug and test, and be more suited to formal verification.”
