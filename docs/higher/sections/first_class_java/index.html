    First-class method #

    By default, imperative languages do not provide a syntax to treat methods as regular values.

    This rules out higher-order methods.

    We will see how Java 8 (2014) introduced a syntax to:

    • refer to a method, and
    • give it a type.

    Method type in Java #

    Java VS Haskell #

    Example. We have seen already that a function $f \colon \mathsf{String} \to \mathbb{Z}$ in Haskell is written:

    f :: String -> Integer
f s = <some code here>

    In Java, this function is written:

    Integer f(String s) {
  <some code here>
}

    A type for a function/method consists of:

    • a type for its input, and
    • a type for its output.

    Example (continued). In Haskell, the function f above has type

    String -> Integer

    Question. What is the type of the function f in Java?

    Java 8 (2014) introduced a syntax (explained below) for the type of a method.

    Example (continued). Since Java 8, the function f above has type

    Function<String, Integer>

    This allows us to write higher-order methods:

    Example (continued). We can define in Java a method:

    String g(Function<String, Integer> f, String s)

    We may call this method g as follows:

    g(x -> x.length(), "Hello")

    The equivalent of g in Haskell is an uncurried function:

    g :: ((String -> Integer), String) -> String

    Currying #

    Java method types also let us write functions in curried form.

    Example (continued). In Haskell, the curried form of the function g above is usually preferred:

    g :: (String -> Integer) -> String -> String

    We can call this function with

    g f

    which evaluates to a function with type

    String -> String

    Similarly, we can define the curried form of g in Java:

    Function<String, String> g(Function<String, Integer> f)

    and call it with

    g(f)

    Convention. Currying is useful in some contexts, but is not the norm in Java (as opposed to Haskell). One reason is syntactic.

    Example. The Haskell function

    h :: Integer -> Integer -> Bool -> String -> String

    is written in Java

    Function<Integer, Function<Boolean, Function<String, String>>> h(Integer i)

    which is not easy to read. In contrast, the uncurried form of this method is

    String h(Integer i1, Integer i2, Boolean b, String s)

    Functional interface #

    Notation #

    Notation. For readability, we may use an abstract notation to describe the type of a Java method, inspired by mathematical functions.

    Examples. We may use the type:

    • Integer $\times$ Integer $\to$ Integer for a method that takes two Integers and returns another,
    • NPC $\times$ Integer $\to$ NPC for a method that takes an NPC and an Integer (in that order), and returns an NPC,
    • NPC[] $\to$ void for a method that takes an array of NPCs and returns nothing.

    Definition #

    A method type in Java is described with a so-called functional interface.

    Syntax. A Java functional interface is an interface with a single (standard) method.

    Note. “Standard” here stands for a method that does not have the default or static qualifier (a “standard” interface method is sometimes called “abstract”).

    Example. The following is a functional interface, which describes the type NPC $\times$ NPC $\to$ Boolean.

    interface NPCPerNPCToBoolean {

    Boolean execute(NPC c1, NPC c2);
}

    Any method with type NPC $\times$ NPC $\to$ Boolean automatically implements this functional interface.
    For instance, the method

    Boolean hasLessXPThan (NPC c1, NPC c2){
    return c1.xp <= c2.xp;
}

    automatically implements this interface.

    An instance of a functional interface (which is a method) can be executed by calling the (only) method declared in the interface, as an instance method.

    Example (continued).

    hasLessXPThan.execute(new Unicorn(3), new Butterfly(4));

    is equivalent to

    hasLessXPThan(new Unicorn(3), new Butterfly(4));

    This is useful when a variable stands for a method.

    Example (continued).

    NPCPerNPCToBoolean f = getAMethod();

f.execute(new Unicorn(3), new Butterfly(4));

    Syntax. The name of a functional interface is irrelevant, as well as the name of its (unique) method.

    Example. The following functional interface also describes the type NPC $\times$ NPC $\to$ Boolean.

    interface Just {

    Boolean because(NPC we, NPC can);
}

    Syntax. A functional interface can be annotated with @FunctionalInterface.
    This is considered good practice:

    • for documentation, and
    • because it prevents compilation if the interface has more than one (non-default) methods.

    Example.

    @FunctionalInterface
interface NPCPerNPCToBoolean {

    Boolean execute(NPC c1, NPC c2);
}

    Native functional interfaces #

    Java provides native (generic) functional interfaces for some frequent method types.

    We highlight here a few of them:

    Function #

    The functional interface

    Function<T,R>

    stands for the type T $\to$ R.

    Its method is called apply.

    Example. A method with type NPC $\to$ Integer implements the functional interface

    Function<NPC, Integer>

    Example. Here is a non-generic and uncurried version of the higher-order function map:

    List<Integer> myMap(Function<NPC, Integer> f, List<NPC> cs){
    List<Integer> outputList = new ArrayList<>();
    for (NPC c : cs){
        outputList.add(f.apply(c));
    }
    return outputList;
}

    Hint. We will see a more concise syntax for (a generic version of) this function in the section dedicated to Java Streams.

    BiFunction #

    The functional interface

    BiFunction<T, U, R>

    stands for the type T $\times$ U $\to$ R.

    Its method is also called apply.

    Example. A method with type NPC $\times$ NPC $\to$ Integer implements the functional interface

    BiFunction<NPC,NPC,Integer>

    BinaryOperator #

    The functional interface

    BinaryOperator<T>

    stands for the type T $\times$ T $\to$ T.

    Its method is also called apply.

    This is a specific case of Bifunction, where all three types coincide (in other words, BinaryOperator<T> extends BiFunction<T,T,T>).

    Example. A method with type Integer $\times$ Integer $\to$ Integer implements the functional interface

    BinaryOperator<Integer>

    Consumer #

    The functional interface

    Consumer<T>

    stands for the type T $\to$ void.

    Its method is called accept.

    Example. A method with type NPC $\to$ void implements the functional interface

    Consumer<NPC>

    Supplier #

    The functional interface

    Supplier<T>

    stands for the type void $\to$ T.

    Its method is called get.

    Example. A method with type void $\to$ NPC implements the functional interface

    Supplier<NPC>

    Predicate #

    The functional interface

    Predicate<T>

    stands for the type T $\to$ Boolean.

    Its method is called test.

    Note. Predicate<T> is syntactic sugar for Function<T,Boolean>, but their methods have different names (test and apply respectively).

    Method reference #

    Syntax. In order to refer to a method, we can use a method reference, written

    • <ClassName>::<methodName> for a static method,
    • <objectVariable>::<methodName> for an instance method,
    • <ClassName>::new for a constructor.

    Example. The static method

    double ceil(double x)

    of the class Math intuitively rounds up x to the nearest integer.

    Now assume a function

    List<Double> myMap(Function<Double, Double> f, List<Double> xs)

    We can pass the method ceil to myMap as follows:

    myMap(Math::ceil, List.of(2.0,3.1));
