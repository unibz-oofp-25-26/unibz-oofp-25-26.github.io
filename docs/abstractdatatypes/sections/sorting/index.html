    Sorting #

    Sorting a collection (of values or objects) is needed in a variety of scenarios.

    Sorting criterion #

    In order to sort a collection of values (resp. objects), we need a sorting criterion, i.e. a way to compare two values (resp. objects).

    Some datatypes come with a natural way to compare two values.

    Example. Two real numbers can be compared w.r.t. $\le$.

    Some datatypes have several natural sorting criteria.

    Example. Two strings may be compared alphabetically from left to right (in English, Hindi, Russian, etc.) or from right to left (in Arabic, Hebrew, Persian, etc.).

    Question. What about objects, i.e. which sorting criterion can we use to sort a collection of:

    • people,
    • cities,
    • sets,
    • trees,
    • etc. ?

    Answer. Any total preorder (i.e. a total, reflexive and transitive binary relation) can be used to sort a collection.

    Which of the following are total preorders (i.e. can be used as a sorting criterion)?

    1. Compare two real numbers w.r.t. to their absolute value.
    2. Compare two people w.r.t. to their age.
    3. Compare two people w.r.t. to their age and size, i.e.

    $\qquad \qquad p_1 \preceq p_2$ iff $\big(p_1.{\text{age}} \le p_2.{\text{age}}$ and $p_1.{\text{size}} \le p_2.{\text{size}}\big)$.

    1. Compare two people w.r.t. to their age or size, i.e.

    $\qquad \qquad p_1 \preceq p_2$ iff $\big(p_1.{\text{age}} \le p_2.{\text{age}}$ or $p_1.{\text{size}} \le p_2.{\text{size}}\big)$.

    1. Compare two people w.r.t. to their age, and then their size (if they have the same age).
    2. Compare two points w.r.t. to their X coordinate, and then their Y coordinate (if they have the same X coordinate).
    3. Compare two trees w.r.t. to their number of nodes.
    4. Compare two sets w.r.t. to set inclusion.
    5. Compare two sets of characters by first sorting each set (by alphabetical order), and then comparing the two resulting sequences lexicographically.
      1. is a partial preorder,
      1. is not transitive, and
      1. is a partial order.

    The others are total preorders.

    To see why 4. is not transitive, consider the example:

    Person Age Height
    Alice 24 175
    Bob 25 160
    Carol 21 170

    Properties of sorting algorithms #

    Sorting algorithms have been extensively studied.

    Disclaimer. We will not cover sorting algorithms this semester (with one exception), because this is part of another course of the bachelor. We only highlight some of their properties.

    Stability #

    Definition. A sorting algorithm is stable if it preserves the initial order (in the input collection) of two elements that are equivalent w.r.t. the sorting criterion.

    Example. Consider an array [c1, c2] of type NPC[], where (the objects referenced by) c1 and c2 both have XP 2.

    Let us sort the elements of this array by XP.

    • A stable sorting algorithm outputs [c1, c2].
    • A non-stable sorting algorithm may output [c1, c2] or [c2, c1].

    In place #

    Definition. A sorting algorithm is in place if it does not use additional data structures (precisely, if it only requires a fixed amount of extra memory).

    in Java #

    Comparing #

    Java provides several native ways to define a sorting criterion (a.k.a. total preorder). We illustrate here one of them.

    Comparable #

    A class T can implement the interface Comparable<T>.

    Example. We can create a class City that implements Comparable<City> as follows:

    public class City implements Comparable<City> {
    ...
}

    The interface Comparable<T> specifies a single instance method:

    int compareTo (T otherObject);

    Usage. The method compareTo should define a total preorder $\preceq_o$ over instances of T, as follows.

    o1.compareTo(T o2)

    should evaluate to

    • a negative integer if o1 $\prec_o$ o_2 (i.e. if o1 $\preceq_o$ o2 and o2 $\not\preceq_o$ o1),
    • 0 if o1 $=_o$ o2 (i.e. if o1 $\preceq_o$ o2 and o2 $\preceq_o$ o1),
    • a positive integer otherwise.

    Example (continued). Here is an implementation of the method compareTo where cities are compared by zip code (in increasing order):

    public class City implements Comparable<City> {  // `City` implements the interface `Comparable<City>`

    public String name;
    public int zipCode;

    public City(String name, int zipCode) {
        this.name = name;
        this.zipCode = zipCode;
    }

    @Override
    public int compareTo(City otherCity) {
        return zipCode - otherCity.zipCode; // this implementation satisfies the constraints above
    }
}

    Default implementations #

    Some native Java classes already implement Comparable, in the expected way.
    These include Integer, String (left to right alphabetical order), etc.

    Comparator methods are also available for primitive types (like int, or bool).

    Example (continued). Integer has a static method compare(int x, int y) that behaves like compareTo. So the method compareTo in the example above can alternatively be written:

    public class City implements Comparable<City> {
  ...

    @Override
    public int compareTo(City otherCity) {
        return Integer.compare(zipCode, otherCity.zipCode);
    }
}

    Warning. The method compareTo is implicitly used by some of Java’s native data structures (e.g. TreeSet).

    So it is usually recommended to implement compareTo in such a way that it “complies” with equals (and hashCode).
    This means that

    $\qquad \qquad $o1.compareTo(o2) should return 0 iff o1.equals(o2) returns true.

    If you want to use a sorting criterion that does not satisfy this constraint, then we recommend using a Comparator instead.

    Comparator #

    To go further. The Java interface Comparator<T> is a more flexible (albeit more verbose) solution to specify a sorting criterion.

    Intuitively, a Comparator stands for a total preorder.

    This interface also provides convenient (default) methods. For instance, the method thenComparing returns the lexicographic product of two comparators.

    Sorting #

    Java provides several methods to sort an array or collection. We only highlight one of them.

    Sorting an array #

    Syntax. The class Arrays provides a static method

    void sort(T[] array)

    that sorts an array.

    Example.

    int[] integers = new integer[]{2,1,3,2};
Arrays.sort(integers)

// outputs [1,2,2,3]
System.out.println(Arrays.toString(array));

    This method sorts according to the so-called “natural ordering” for type T.

    The “natural ordering” for a Java type T is:

    • the one expected if T is a primitive types (like int), boxed types (like Integer) or one of a few other native types (like String or Date); a list can be found here,
    • the one defined by T.compareTo if T implements Comparable,
    • underspecified otherwise.

    Example(continued).

    City bologna =  new City("Bologna", 40100);
City trento =  new City("Trento", 38121);
City[] cities = new City[]{bologna, trento};

// after this, the array 'cities' contains [trento, bologna],
// due to our implementation of `compareTo` for `City
Arrays.sort(cities);

    To go further. The static method

    sort(T[] array, Comparator<? super T> c)

    of the class Arrays is similar, but it uses the input comparator to sort the array.

    Algorithms. For an array with primitive type (like int[]), Arrays.sort uses (a version of) the Quicksort algorithm, even though Quicksort is not stable. Stability is irrelevant for these types: for instance, permuting the elements of the array [5,5] yields the same array.

    In contrast, for an array with (arbitrary) reference type (like City[]), Arrays.sort uses (a version of) the MergeSort algorithm, which is stable.

    Sorting a list #

    Syntax. The class Collections provides a static method

    void sort(List<T> list)

    whose behavior is analogous to Arrays.sort.

    In particular, it also guarantees stable sorting.

    Example(continued).

    City bologna =  new City("Bologna", 40100);
City trento =  new City("Trento", 38122);
List<City> cities = new LinkedList();
cities.add(bologna);
cities.add(trento);

// after this, the array 'cities' contains [trento, bologna],
// due to our implementation of `compareTo` for `City
Collections.sort(cities);

    To go further. Like Arrays.sort, the method Collections.sort has a variant that takes a Comparator as additional argument.
