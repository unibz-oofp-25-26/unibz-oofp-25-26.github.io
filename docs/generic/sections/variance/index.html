    Variance #

    Notation (reminder). We use $A \sqsubseteq B$ to indicate that $A$ is a subtype of $B$.

    Observation. In some cases, subtyping between parameterized types is intuitively determined by subtyping between their parameters.

    Example. A set of integers is a set of numbers.

    Some object-oriented languages (like Java or C#) let us use this property to extend subtyping to generic types. We will focus here on Java.

    Definition #

    Covariance #

    Example. Consider the generic type Set<T>

    Intuitively:

    • Set<Integer> is a subtype of Set<Number>
    • Set<Butterfly> is a subtype of Set<NPC>
    • etc.

    More generally, if A is a subtype of B, then Set<A> is a subtype of Set<B>.

    When this holds, the generic type is said to be covariant.

    Definition. A generic type Gen<T> is covariant if for any types A and B,

    $\qquad$ A $\sqsubseteq$ B implies Gen<A> $\sqsubseteq$ Gen<B>

    Contravariance #

    What intuitively holds above for sets does not seem to hold for all generic types.

    Example. Recall the Java functional interface Predicate<T>, which stands for functions that return a Boolean value.

    As we saw in the dedicated section, if A is a subtype of B, then

    $\qquad$ B $\to$ Boolean

    should intuitively be a subtype of

    $\qquad$ A $\to$ Boolean

    So in this case, we want to treat Predicate<B> as a subtype of Predicate<A>.

    Definition. A generic type Gen<T> is contravariant if for any types A and B,

    $\qquad$ A $\sqsubseteq$ B implies Gen<B> $\sqsubseteq$ Gen<A>

    Invariance #

    Definition. A generic type is invariant if it is neither covariant nor contravariant.

    Combinations #

    If a generic type has several type variables, then it may be independently covariant, contravariant or invariant in each of them.

    Example. Recall the Java functional interface Function<T,R>, which stands for functions with type

    $\qquad$ T $\to$ R

    As we saw in the dedicated section, if X1 is a subtype of X2 and Y1 is a subtype of Y2, then

    $\qquad$ X2 $\to$ Y1

    should intuitively be a subtype of

    $\qquad$ X1 $\to$ Y2

    So in this case, we want like to treat Function<T,R> as:

    • contravariant in T, and
    • covariant in R.

    Implementation #

    As the previous examples show, some generic types are naturally covariant and/or contravariant in (some of) their type variables.

    Some languages (like C# or Kotlin) provide a syntax to declare that a generic type should always be treated by the compiler as covariant (resp. contravariant, invariant or a combination of them).

    Java does not support this.

    in Java #

    Array #

    Terminology. Java arrays predate (by almost 10 years) the introduction of generics. So technically, “array” is not a generic type in Java. Still, it can be viewed of as a “generic-like” type.

    Warning. Java arrays are covariant.

    Example. The following Java program compiles:

    Number[] anArray;
Integer[] anotherArray = new Integer[]{};
anArray = anotherArray;

    Generic type #

    Constraint. Java generic types are invariant.

    Example. The following Java program does not compile, because the generic type List<E> is invariant:

    List<Number> aList;
List<Integer> anotherList = new ArrayList<>();
aList = anotherList;

    So a Java generic type cannot be declared as covariant or contravariant.

    Instead, each time we use a generic type, we can specify that it should be interpreted by the compiler as covariant (resp. contravariant, or some combination of these).

    Covariance #

    Syntax. We saw earlier that the keyword extends specifies an upper bound on the value of a type variable. We can also use it with a ? to allow covariance.

    Example. The following Java program does compile:

    List<? extends Number> aList;
List<Integer> anotherList = new ArrayList<>();
aList = anotherList;
    Contravariance #

    Syntax. The keyword super is symmetric to the keyword extends. It can be used to allow contravariance (in combination with ?).

    Example. Recall the method filter of the interface Stream<T>.

    It takes as argument a Boolean function (a.k.a. Predicate in Java), which maps each element of the stream to true or false.

    We have seen above that Predicate is intuitively contravariant.
    Accordingly, the signature of filter is

    Stream<T> filter(Predicate<? super T> predicate)

    It accepts as argument any method with type Predicate<V> such that T $\sqsubseteq$ V.
    So the following program compiles:

    List<Integer> integers = getIntegers();
Stream<Integer> myStream = integers.stream()
                  .filter(MyClass::myPredicate);

public class MyClass {

    static boolean myPredicate(Number number) {
        ...
    }
}
    Combinations #

    Example. Recall the method map of the interface Stream<T>.

    It takes as argument a function from T to R (a.k.a. Function in Java), and returns a Stream<R>

    As we saw earlier, the generic type Function is intuitively:

    • contravariant in its first parameter, and
    • covariant in its second parameter.

    Accordingly, the signature of map is:

    <R> Stream<R> map(Function<? super T, ? extends R> mapper)
